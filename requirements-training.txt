# Pinned Python dependencies for the PyTorch training/aggregation worker.
#
# These are installed into an isolated venv (/opt/training-venv) by the
# runtime stage of the Dockerfile. Pinning (==) — instead of the previous
# unpinned `pip3 install torch numpy` — makes the image build reproducible
# and removes the supply-chain risk of silently pulling a newer (possibly
# compromised or behaviorally-divergent) release on every rebuild.
#
# The training scripts (scripts/diloco_*.py, scripts/train_*.py) hard-disable
# CUDA and run CPU-only, so the default CPU wheels from PyPI are sufficient.
#
# Version policy (design section 4.2 — what a torch bump must re-validate):
#
# The DiLoCo consensus key is the set of SCALAR INVARIANTS the aggregate
# executor emits: avgGradientNorm, velocityNorm, and the accepted/rejected
# peer sets. Those scalars are bit-stable given pinned inputs, and the
# coordinator reaches consensus on them with a tolerance, NOT on any adapter
# byte hash. The adapter PICKLE bytes / adapter sha256 are NON-deterministic
# across runs (torch tensor pickling embeds storage metadata) and are
# informational only. So when bumping torch, the re-validation target is
# SCALAR-INVARIANT STABILITY (avgGradientNorm / velocityNorm / accept-reject
# sets bit-equal across runs and across the old/new torch) — NOT an adapter
# sha256. Run `pnpm --filter @synapseia-network/node test:diloco-compat`
# (authoritatively inside the pinned /opt/training-venv) before shipping.
#
# Where torch is and is NOT needed:
#   - COORDINATOR: verifies each uploaded submission blob by its sha256 +
#     Ed25519 signature for INTEGRITY and AUTH only. It never deserializes
#     torch tensors, so the coordinator needs NO torch pin.
#   - NODES (training + aggregating): every node MUST share this exact `==`
#     pin so the scalar invariants are computed identically across the swarm.
#     Divergent torch on two nodes can drift the invariants past the coord's
#     tolerance and split consensus. The `==` pin is what enforces that
#     uniformity (and keeps the image build reproducible / supply-chain safe).
#
# To regenerate with hashes, run `pip hash` or use
# `pip-compile --generate-hashes` against this file in CI.
#
# Pin history (PyPI drift + Python 3.14 / cp314):
#   torch==2.5.1 / numpy==2.1.3 → both AGED OUT of PyPI. The default PyPI
#   index now serves only torch {2.9.0, 2.9.1, 2.10.0, 2.11.0, 2.12.0}
#   (2.5.1 and the 2.6.0 the installer briefly used are gone), and neither
#   2.5.1 nor 2.6.0 ever shipped a cp314 wheel. A Python-3.14 node (e.g.
#   node-kike, venv Python 3.14.5) therefore could not install the pinned
#   torch and booted WITHOUT torch / training caps:
#     ERROR: Could not find a version that satisfies the requirement
#     torch==2.6.0 (from versions: 2.9.0, 2.9.1, 2.10.0, 2.11.0, 2.12.0)
#   New pin: torch==2.9.1 — the OLDEST stable still served (least
#   bleeding-edge) that ships a cp314 wheel on the default/cpu index AND
#   the cu128 NVIDIA index. This MUST stay byte-identical to the version
#   resolved by src/utils/install-deps.ts `selectTorchSpec` (the ADR
#   requires every training node share the exact `==` pin) — see
#   docs/adr-0001-diloco-torch-pin-policy.md.
#   numpy==2.1.3 likewise has NO cp314 wheel (numpy cp314 wheels start at
#   2.3.2); bumped to numpy==2.3.2 (lowest cp314-capable) so the pinned
#   venv is installable on Python 3.14.
torch==2.9.1
numpy==2.3.2
